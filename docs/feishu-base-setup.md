# Feishu Base Setup / 飞书多维表格建表规范

> 这份文档告诉你：要在自己的飞书多维表格里跑通这个 Agent，需要建哪些表、每张表要有哪些字段。
>
> 表名和字段名是参考命名，你可以改名，但**字段语义**要对应得上 prompts 和 schemas 里的引用。

---

## 总览：8 张表

| 表号 | 表名 | 角色 | 必备 |
|------|------|------|------|
| 01 | `01_房源信息` | 字典：房源主数据 | ✅ |
| 02 | `02_渠道信息` | 字典：订单渠道 | ✅ |
| 03 | `03_交易平台信息` | 字典：交易平台 | ✅ |
| 04 | `04_支付方式信息` | 字典：支付方式 | ✅ |
| 05 | `05_分类规则` | 字典：一级/二级科目 | ✅ |
| 06 | `06_订单台账001` | 账本：房晚订单 | ✅ |
| 07 | `07_运营收支001` | 账本：运营类支出/收入 | ✅ |
| 08 | （可选）证据归档 | 原始截图链接归档 | ⛔️ 可省 |

`001` 后缀是为了支持"团队隔离 / 多业主分账"——一个飞书多维表格 base 内可以放 `06_订单台账001`、`06_订单台账002` 等多套账本，对应不同团队。单业主用户用 `001` 即可。

## 01_房源信息

| 字段 | 类型 | 说明 |
|------|------|------|
| `房源` | 文本（主键展示） | 房源唯一标识，建议格式："示例路A弄N号" |
| `城市` | 文本 | 例：上海 |
| `区域` | 文本 | 例：黄浦区 |
| `项目` | 文本 | 同一物业群组的标识 |
| `房号` | 文本 | 同一项目下的房号 |
| `渠道` | 单选/多选 | 主要渠道，关联 `02_渠道信息` |
| `房源ID` | 文本（自动生成或手填） | 用于跨表关联 |
| `状态` | 单选 | 在营 / 暂停 / 已退租 |
| `地标别名` | 文本 | 用于 LLM 识别时的模糊匹配（如某新村地名指向具体房源） |

## 05_分类规则

科目字典，决定运营类记录的"一级科目 / 二级科目"。

| 字段 | 类型 | 说明 |
|------|------|------|
| `一级科目` | 文本 | 例：耗材 / 维修 / 清洁 / 公用事业 / 营销 |
| `二级科目` | 文本 | 例：纸巾、洗发水、灯泡、家政、电费、水费 |
| `关键词` | 多选 | LLM 推断时的命中关键词 |
| `是否启用` | 复选框 | 软删用 |

**重要**：二级科目要"挂在"一级科目下（同一行）。Agent 在卡片上做"二级科目下拉"时会按当前选中的一级科目过滤。

## 字段命名硬约定

如果你想改字段名，请保留以下"语义键"，因为 prompts 和 schemas 是按这些键引用的：

- `账务日期` / `摘要` / `金额` / `房源` / `数量`
- `一级科目` / `二级科目`
- `交易平台` / `支付方式` / `收支方向` / `结算状态`
- `订单名称` / `渠道` / `间夜` / `入住日期` / `离店日期`
- `备注` / `订单摘要` / `订单ID` / `运营记录ID` / `证据文本` / `证据附件`

改了之后你需要同步修改 prompts/ 下的提示词。

## 多团队/多业主隔离

如果你管的不只一家民宿，建议：

- 每个业主独立一个飞书多维表格 base
- 表号后缀（`001` / `002`）用于同 base 下的子账本
- Agent 启动时通过群聊 ID 或租户 ID 路由到对应 base + 后缀

不要在同一张表里塞多个业主——查询、汇总、隐私都会出问题。
