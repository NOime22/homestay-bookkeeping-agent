# Lessons Learned — 飞书卡片 + LLM 工作流踩坑笔记

> 这些都是真实做这个 Agent 过程中踩出来的坑。飞书官方文档要么没写，要么藏得很深，要么报错和原因严重不对应。希望能帮后来者省下几天调试。
>
> The Feishu docs are sparse on edge cases. Below are real production gotchas — hopefully they save you a few days.

---

## 1. 卡片渲染 / Card Rendering

### 1.1 `select_static` 必须放在 `action` 块里，否则不渲染

**症状**：卡片发送成功（API 200 OK，没有报错），但下拉框在客户端**完全不显示**——飞书 App 上看是一片空白。

**原因**：`select_static` 在飞书 CardKit 里被设计为"交互动作组件"，必须挂在 `action` 块（按钮、动作行）下。直接放进 `div` / `column` 的 elements 数组会被静默忽略。

**修复**：
```json
{
  "tag": "action",
  "actions": [
    { "tag": "select_static", "placeholder": ..., "options": [...] }
  ]
}
```

**关键词**：select_static 不显示 / 下拉框消失 / interactive 卡片空白

---

### 1.2 用 `form` 包裹的 `select_static` 会发出去但渲染不稳定

**症状**：用 `form` 把多个下拉打包成一个表单，API 返回成功，部分客户端能显示部分不行，且回调里 `form_value` 字段恒为 `null`。

**结论**：在当前 CardKit 实测下，**不要用 `form` 包多个交互组件**。每个下拉单独放 `action` 块。

---

### 1.3 `input` 和 `date_picker` 在此场景不可靠

**症状**：`input` 组件的回调中拿到的 value 形态和文档描述不符；`date_picker` 在某些客户端不触发回调。

**变通**：对金额、日期这类"易错数字字段"，**不要做卡内自由输入**。改用"卡片显示在文本区 + 用户点'有文字修改'按钮 + 回复文字模板"的方式。详见 [`prompts/text-edit-merge.md`](prompts/text-edit-merge.md)。

---

## 2. 模板卡 / Template Card

### 2.1 `initial_option` 字段会报 230099

**症状**：尝试给 `select_static` 预选一个默认值（用 `initial_option`），API 直接返回 **230099** 参数错误。

**原因**：飞书 CardKit 在模板卡里**不接受 `initial_option`**——预选值只能通过 `initial_index` 指定。

**修复**：把"我想预选某个值"翻译成"我想预选第几项"，传 `initial_index`。

---

### 2.2 `initial_index` 是 **1-indexed**，不是 0-indexed

**症状**：本来想预选第一个选项（你以为 `initial_index=0`），结果客户端显示的是 placeholder，没预选；或者预选了"上一项"。

**原因**：飞书模板卡的 `initial_index` 从 **1 开始数**，不是程序员习惯的 0。

**修复**：
- 想预选第 1 项 → `initial_index: 1`
- 想预选第 N 项 → `initial_index: N`
- **不要**用 `options.find(x => x.value === preset)` 的索引直接传——记得 +1

---

### 2.3 模板版本必须显式管理

模板卡有版本号（如 `1.0.9`）。发送时若不指定版本，飞书会用"草稿版"，导致：
- 测试环境正常、生产环境布局错乱
- 你在网页编辑器里改了卡，但客户端看到的还是旧版

**修复**：把 `template_id` 和 `template_version_name` 都写进配置，一次只跑一个明确版本：
```yaml
bookkeeping_card_template_id: <YOUR_CARD_TEMPLATE_ID>
bookkeeping_card_template_version_name: "1.0.9"
```

## 3. 回调 / Callback

### 3.1 下拉转圈不停的根因：回调没返回 card + Toast

**症状**：用户点了一下下拉，UI 上转圈圈不停（loading 永远不结束），但服务端日志已经收到回调、也处理完了。

**原因**：飞书客户端在等回调返回**两样东西**才会停止 loading：
1. 刷新后的 card payload（template_id + 新 variables）
2. 一个 `toast`（可以是 hidden 的，但必须有）

只回 200 OK / 只返回新 card / 只返回 toast 都不够。

**修复**：回调响应一定要同时包含：
```json
{
  "toast": { "type": "info", "content": "" },
  "card": {
    "type": "template",
    "data": {
      "template_id": "<YOUR_CARD_TEMPLATE_ID>",
      "template_variable": { ...刷新后的全量变量... }
    }
  }
}
```

---

### 3.2 `form_value` 在 CardKit 模板卡里**恒为 null**

**症状**：你按官方文档去读 `event.action.form_value`，发现永远是 `null`，无论用户改了多少个下拉。

**原因**：CardKit 模板卡的回调用的不是 form 提交模型。每个组件的值在回调 payload 里的位置不同：
- `select_static` 的当前选项 → `event.action.option`
- `input` 的当前文本 → `event.action.input_value`（**不是** `form_value.input`，**不是** `value`）
- `select_person` 的当前选择 → `event.action.option`

**修复**：写一个解析函数，按 `event.action.tag` 分支取值，不要寄望于一个统一的 `form_value`。

---

### 3.3 一个用户消息只能用"卡片"回复，不能再发普通文本

**症状**：用户发了张账单截图，你想先发一段"识别中..."文本，再发卡片——结果只有第一条出现，卡片消失了；或者反过来。

**结论**：飞书群里同一条用户事件下，**只能发一种结构化回复**。你只能选：
- 发一张卡片（推荐）
- 发一段文本（降级）

**修复**：所有阻塞、错误、提示都用"内嵌在卡片里的文字行"表达，不要试图用"卡片 + 旁白文本"组合。

---

## 4. 数据 / 字典

### 4.1 截图证据优先于文本证据

**症状**：用户发"截图 + 文字描述"组合消息，文字说"这是订单收入"，但截图明显是采购小票。如果你按文字判型，会归错类。

**结论**：**有截图存在时，以截图判型；纯文本时，以文本判型。** 文字只用来补充字段（金额、日期、备注），不参与一二级分类。

---

### 4.2 摘要要短，6-10 字内

**症状**：让 LLM 写摘要，它默认写成"在拼多多购买的家用抽象壁画一幅用于民宿装饰"。

**修复**：在 prompt 里硬性约束 `≤10 字`，只保留核心品名："抽象壁画"。备注字段默认留空给用户。

---

### 4.3 平台词典里"闪购"要映射到"淘宝"

**症状**：用户发淘宝闪购的支付截图，LLM 给"交易平台"填了"闪购"，但你的字典里没有这一项 → 下拉对不上 → 写账失败。

**修复**：把这类常见同义词在 prompt 里硬绑：闪购=淘宝、便利店=线下、外卖=美团等。词典本身保持业务面的命名。

---

### 4.4 房源识别：地标 + 业务别名

**症状**：截图里只有"示例新村地下车库停车费"，没写完整地址。如果没有兜底规则，房源字段会留空。

**修复**：在 prompt 里维护一份"地标 → 房源"映射；并约定"如果该地段只有一个房源，自动指向唯一房源"。无法匹配时**留空追问**，不要瞎填。

---

## 5. 文字修改回路 / Text-Edit Loop

### 5.1 "有文字修改"必须能绑定回原 candidate

**症状**：用户点了"有文字修改"，回复了"金额：50"，但因为 reply_to_message_id 丢失，系统不知道这条文字属于哪张卡，文字被忽略。

**修复**：发送文字模板时记录 `text_edit_prompt_message_id`，并提供两条绑定路径：
1. 优先：用 `reply_to_message_id` 精确绑定
2. 降级：同一 chat + 同一 actor 下只有一张 pending draft 时，绑定到那张

详见 [`adapters/feishu.md`](adapters/feishu.md)。

---

### 5.2 合并文字时只接受非空 `字段：值` 行

**症状**：用户回复模板时漏写了几个字段（只填了金额），如果你严格要求"必须填满才合并"，体验极差。

**修复**：解析时只取非空行、忽略格式错误行、空白行直接丢。允许用户"只改一个字段"。

---

### 5.3 文字合并后必须刷新真卡，不能退化成文本

**症状**：用户文字修改后，系统回了一段"已收到您的修改：金额=50，日期=2026-06-23..."的普通文本，没刷新卡片。

**结论**：文字合并后**必须再发一张刷新后的真 interactive card**，让用户能继续点"确认记账"或再次修改。退化成文本就走不到最终确认。

---

## 6. 写账 / Writeback

### 6.1 证据图通过云盘上传转链接，不要直接附附件

**症状**：你想把识别用的原图作为附件放进飞书多维表格的"证据附件"列，结果体积大、网络慢、有时直接失败。

**修复**：用 lark-cli 或飞书 OpenAPI 先把图传到云盘（drive），拿到 `file_token` 之后构造 `https://<base_host>/file/<token>` 形式的链接，写进表格的链接列。比附件列稳定得多。

---

### 6.2 日期字段：未识别时用消息时间戳兜底

**症状**：截图里没有清晰日期（比如商品列表页），LLM 漏填账务日期，导致写账时表格报错。

**修复**：识别阶段就先用消息发送时间作为默认 `账务日期`，并在卡片文本区**显式显示**这个默认值，让用户能在卡里修正。

---

## 7. Agent 行为 / Agent Behavior

### 7.1 启动消息只进角色，不要立刻发卡

**症状**：每次会话启动时给 Agent 发一段"切换到记账身份"的系统消息，Agent 会顺手就发一张空白确认卡或追问。

**修复**：识别启动消息为"角色切换"语境，**只更新会话状态，不产生任何对外消息**，等到用户真正发账务证据时才走识别流程。

---

### 7.2 LLM 主模型要支持"原生看图"，不要降级 OCR

**症状**：用 OCR + 文本理解的两段式 pipeline，截图识别率显著下降——发票上的中文金额、手写备注、表格结构都识别不准。

**修复**：用支持 vision 的模型直接看图。把"主模型原生看图"作为这个 Agent 的一类验收标准，回归测试时要显式验证不是退化成了 OCR/文本路径。

---

## 8. 通用反模式 / Anti-patterns

不要做的事：

- ❌ 让用户在飞书里手动声明"这是运营类还是订单类"——Agent 应该自己判断
- ❌ 用普通文本伪装成卡片（"请回复 1/2/3 选择..."）——这不是真交互
- ❌ 把 raw OCR 文本 / JSON dump 直接发到群里
- ❌ 自动给"备注"字段塞长描述——留空，给用户决定
- ❌ 给运营类记录强行关联订单——可留空就留空
- ❌ 在记账主链路不可用时，**静默改走**另一条临时发卡路径——必须明确报错
