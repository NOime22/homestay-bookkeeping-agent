# Text-edit Merge Prompt

> 配套规则见 [`../WORKFLOWS.md`](../WORKFLOWS.md) Stage C/D 与 [`../CARD_CONTRACTS.md`](../CARD_CONTRACTS.md) 第 5 节。

## 1. 发送的文字模板

当用户点击卡片上的 `有文字修改` 按钮，宿主**只发一条**极简文字模板：

```
金额：
日期：
订单摘要：
备注：
```

要点：
- 不要加"请按格式回复"等说明文字
- 不要加示例值（避免误删占位符）
- 顺序固定，便于用户填空
- 模板消息发送后，记录 `text_edit_prompt_message_id`，并把 draft 置为 `awaiting_text_reply=true`

## 2. 用户回复的解析规则

接收用户文字回复后：

### 2.1 行级解析

按换行符切，每行尝试解析为 `字段：值`。中文冒号 `：` 和英文冒号 `:` 都接受。

```
金额：50.5
日期: 2026-06-23
订单摘要：
备注：客户加价
```

### 2.2 合并策略

| 行类型 | 处理 |
|--------|------|
| 非空 `字段：值` | 合并进 `edited_fields[字段] = 值` |
| 空值行（如 `订单摘要：`） | 忽略，不清空原值 |
| 解析失败的行（无冒号、字段名不在白名单） | 忽略 |
| 完全空白行 | 忽略 |

### 2.3 字段白名单

仅以下字段允许通过文字修改路径写入：

- `金额` / `日期`（即 `账务日期`）
- `订单摘要`
- `备注`
- `订单ID`（可选，谨慎）

下拉类字段（房源、科目、平台、支付方式、数量、收支方向、结算状态）**不接受文字修改**——只能在卡内下拉里改。

### 2.4 值规范化

- `金额`：去掉 `¥` `￥` `元` `RMB` 等前后缀，保留数字（支持小数点）
- `日期`：接受 `2026-06-23` / `2026/6/23` / `6月23日`（当年默认）等格式，统一归一化为 ISO `YYYY-MM-DD`
- `订单摘要` / `备注`：trim 首尾空白

## 3. 合并后的动作

合并完成后：

1. 清除 `awaiting_text_reply` 状态
2. 直接发送**刷新后的真 interactive card**（不是普通文本回执！）
3. 卡片上必须显示合并后的最新值，按钮维持 `有文字修改` / `确认记账`

## 4. 绑定 candidate 的路径

文字回复怎么知道是哪张 draft 的修改？两条路径，优先级如下：

1. **优先**：用 `reply_to_message_id` 精确匹配。即用户必须"回复"那条文字模板。
2. **降级**：如果飞书没传回 reply 元数据，但当前 chat + actor 下只有**唯一一张** `awaiting_text_reply=true` 的 draft，则绑定到那张。
3. **失败**：以上都不满足 → 当成普通消息处理，不写进任何 draft。

## 5. 反模式

- ❌ 用 input 组件做卡内自由输入（CardKit 数据形态不稳）
- ❌ 让用户回复 JSON（人类不会写）
- ❌ 让用户重新填全部字段（修改一个就该改一个）
- ❌ 合并后只回普通文本（用户走不到确认）
